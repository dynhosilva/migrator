# Módulo cicd — GitHub Actions Generator

Gera workflows GitHub Actions determinísticos a partir da análise do projeto — sem API do GitHub, sem automação de secrets, sem coupling com providers cloud.

## Posição no pipeline

```
analyze → plan → validate → migrate → deploy → cicd → execute → runtime → remote
```

O módulo cicd recebe o `ProjectContext` enriquecido pelas fases anteriores (especialmente `analysis` com stack e `packageJson.scripts`) e escreve os workflows em `.github/workflows/` dentro do `outputDir`.

## Arquitetura

```
src/cicd/
├── types.ts             ← CicdState, CiSummary, ReleaseSummary, CicdIssue
├── workflow-types.ts    ← Tipos GH Actions: GithubWorkflow, WorkflowJob, WorkflowStep…
├── registry.ts          ← CicdRegistry — mesmo padrão dos outros registries
├── index.ts             ← cicdProject(), cicdContext()
├── builders/
│   └── workflow.ts      ← serializeWorkflow() + step builders individuais
└── tasks/
    ├── ci-workflow-generator.ts      ← gera .github/workflows/ci.yml
    └── release-workflow-generator.ts ← gera .github/workflows/release.yml
```

O padrão Registry é idêntico ao de todos os outros módulos: tasks registradas executam em sequência acumulando `Partial<CicdState>`. Adicionar um novo workflow = criar arquivo + registrar.

## Geração de workflows

### ci-workflow-generator

Gera `.github/workflows/ci.yml` com:

- **Trigger:** push e pull_request no branch `main`
- **Matrix:** Node.js `[20, 22]` (fixa — sem coupling com versão do host)
- **npm cache** via `setup-node` action
- **Passos condicionais:** `build` e `test` incluídos apenas se os scripts correspondentes existem em `package.json`

**Exemplo de output (react-vite — sem script test):**

```yaml
# Generated by lovable-migrate — do not edit manually
name: CI
on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main
jobs:
  build:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version:
          - 20
          - 22
    steps:
      - uses: actions/checkout@v4
      - name: Set up Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: npm
      - name: Install dependencies
        run: npm ci
      - name: Build
        run: npm run build
```

### release-workflow-generator

Gera `.github/workflows/release.yml` com:

- **Trigger:** push de tags `v*`
- **Node.js:** versão fixa 20 (LTS estável para releases)
- **`registry-url`** configurado para npm — pronto para ativar publicação real
- **`npm publish --dry-run`** por padrão — seguro sem configuração adicional
- **Zero referências a secrets** — usuário adiciona `NODE_AUTH_TOKEN` manualmente quando pronto

## Builders

`src/cicd/builders/workflow.ts` expõe funções puras que retornam `WorkflowStep`:

```typescript
checkoutStep()                       // uses: actions/checkout@v4
setupNodeMatrixStep(pm)              // uses: setup-node@v4 com matrix
setupNodeReleaseStep(version, pm)    // uses: setup-node@v4 com versão fixa
installStep(pm)                      // npm ci / yarn install / pnpm install / bun install
buildStep(pm)                        // npm run build / yarn build / …
testStep(pm)                         // npm test / yarn test / …
publishDryRunStep()                  // npm publish --dry-run
```

Cada builder recebe `PackageManager` como argumento — os comandos variam por package manager mas a estrutura é idêntica.

`serializeWorkflow(workflow)` usa `yaml.stringify()` com `{ lineWidth: 0, indent: 2 }` + cabeçalho de geração.

## Por que o pacote yaml (eemeli/yaml)?

Três razões concretas:

1. **Template strings são frágeis para YAML.** YAML é indentation-sensitive. Uma string interpolada errada pode produzir YAML inválido silenciosamente, especialmente com valores que contêm `:`, `{`, `#` ou espaços.

2. **Tipagem end-to-end.** Com `GithubWorkflow` como objeto TypeScript serializado, o compilador valida a estrutura antes de qualquer output. Template strings só falham em runtime.

3. **Determinismo garantido.** `yaml.stringify()` serializa chaves na ordem de inserção do objeto. Mesma entrada → mesma saída. Snapshots estáveis, diffs limpos.

**Por que não js-yaml?** O `yaml` (eemeli/yaml v2) usa YAML 1.2, tem zero dependências, e é ativo. js-yaml usa YAML 1.1 (onde `on:` pode ser interpretado como booleano em alguns contextos).

## Por que evitar GitHub API e automação de secrets?

Intencionalmente fora do escopo desta fase:

- **GitHub API** requer autenticação, scope de permissões e acoplamento ao provider. Viola o princípio de zero cloud coupling.
- **Secrets automation** significa escrever tokens em sistemas externos — operação irreversível com blast radius alto fora do controle da engine.
- **OIDC, GitHub Apps, environments** são configurações específicas do repositório que dependem de contexto que o `lovable-migrate` não tem acesso.

A filosofia: gerar o arquivo, deixar o usuário ativar. O CI/CD funciona assim que o usuário faz `git push` — sem pré-requisitos além de um repositório GitHub.

## Determinismo e snapshots

O módulo cicd é **100% determinístico**: mesma análise → mesmo YAML, em qualquer plataforma, em qualquer execução.

Garantias:

- `yaml.stringify()` serializa na ordem de inserção das chaves do objeto
- Step builders retornam objetos literais sem estado mutable
- Nenhum campo usa timestamp, versão do host ou path absoluto
- `CiStepKind`, versões Node e node matrix são constantes no código

Os snapshots em `test/snapshots/cicd.snap` capturam o YAML verbatim. Qualquer mudança no output quebra o snapshot — isso é intencional, não incidental.

**Snapshots estruturais vs host-dependent:**

| Tipo | Exemplo | Estratégia |
|---|---|---|
| Estrutural (invariante) | YAML de workflows, nomes de steps, triggers | `toMatchSnapshot()` direto |
| Host-dependent | Docker disponível, versão Node do host | Normalizado: `<DOCKER_AVAILABLE>`, `<VERSION>` |
| Dinâmico de runtime | durationMs, exitCode, stdout | `normalizeRuntimeLog()` → `<REDACTED>` |

Workflows CI/CD são estruturais — nunca dependem do host.

## Comportamento condicional

O gerador usa `ctx.analysis.packageJson.scripts` para decidir quais steps incluir:

```typescript
const scripts = ctx.analysis?.packageJson?.scripts ?? {};
steps = [
  checkoutStep(),
  setupNodeMatrixStep(pm),
  installStep(pm),
  ...(scripts['build'] ? [buildStep(pm)] : []),
  ...(scripts['test']  ? [testStep(pm)]  : []),
];
```

Projetos sem script `build` não recebem o step de build. Projetos sem script `test` não recebem o step de test. O workflow gerado é sempre válido — nunca falha por um script ausente.

## Testes

`test/integration/cicd.test.ts` tem dois describe blocks:

1. **Contrato do módulo** — garante que `CicdState`, `readiness`, `ci`, `release` e `projectName` estão presentes e corretos
2. **Geração de workflows** — golden tests (snapshot do YAML verbatim) + assertions de invariantes

Invariantes testadas explicitamente:
- Trigger push + pull_request com branch main no CI
- Matrix com Node 20 e 22
- npm cache
- Passos install + build (condicionais ao fixture)
- Ausência de step test (fixture react-vite não tem script test)
- Trigger tag `v*` no Release
- `npm publish --dry-run` presente
- `registry-url` para npm
- Zero `secrets.` e zero `NPM_TOKEN` no arquivo

## Referência de tipos

```typescript
interface CicdState {
  readonly projectName: string;
  readonly outputDir:   string;
  readonly ci:          CiSummary;
  readonly release:     ReleaseSummary;
  readonly readiness:   CicdReadiness;  // 'ready' | 'ready-with-warnings' | 'blocked'
  readonly generatedAt: string;
}

interface CiSummary     { files: GeneratedFile[]; issues: CicdIssue[]; }
interface ReleaseSummary { files: GeneratedFile[]; issues: CicdIssue[]; }

type CiStepKind = 'install' | 'typecheck' | 'test' | 'build'
                | 'publish' | 'docker' | 'health-check';
```

Ver `src/cicd/types.ts` e `src/cicd/workflow-types.ts` para as interfaces completas.
